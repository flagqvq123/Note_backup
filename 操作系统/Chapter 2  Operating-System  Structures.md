# 2.System Calls
	操作系统提供服务的编程接口，就像一类特殊的API。
	系统底层存在一些服务函数供上层调用。这些函数一般功能单一、简单。它是一种更底层的API。

##### 系统调用：
应用层调用一个系统服务API（system call），都会从用户态切换到内核态执行内部的指令。完成后将结果返回并将控制权交还到用户态。

##### 系统调用的类型：
- 进程控制（创建、加载进程，结束、打断进程，……）
- 文件处理（增、删、改、查） 
- 设备处理
- 系统信息
- 通信
- 安全与防护（控制访问权限……）

##### 前端后端(foreground/background)
- 前端：直接给用户交互的界面/服务，具有强力的时间敏感性，所以一般具有非常高的优先级。
- 后端：不直接与用户接触的服务。例如“保存”。后端进程的优先级不一定高，对延时不一定敏感。
- batch processing（批处理）：将后端进程集中处理的方法。

##### ==Linkers and Loaders==
- Static loading 静态加载：一次性将运行程序需要的所有程序和数据加载到内存。
- ==Dynamic loading 动态加载==：操作系统在运行程序时分批次将所需要用的代码加载到内存中。相比静态加载，能够节省内存空间，运行更多的程序。
- Static Linking 静态加载：每个程序独自加载各自的头文件和类。然而不同程序的相同的头文件或者类被反复加载到内存上。
- ==Dynamic Linking 动态链接==：操作系统会评估不同的程序共用哪些头文件、函数、类等，然后会构建出动态加载的库供多个程序使用。这样会拖慢单个文件的速度，还可能产生兼容性问题，但是这样的做法依旧利大于弊。
- 在windows里，这个动态库的脚本叫做dynamic linking library，也即`.dll`文件。Linix内叫script脚本。

##### 定制化操作系统与应用：
	在实质上，操作系统正在变得越来越定制化，不同的硬件所对应的操作系统不同，更加有利于它们发挥各自的优势。相应的，不同操作系统上的应用程序的构建也不同。
	比如在手机上的sleep()的实现也非常不同，手机为了能耗啊做出了大量的操作系统层面的改变。

- 从用户的角度：简单、方便、易上手。`图形化>命令行`
- 从系统的角度：希望将操作系统变得易与开发的方向。`模块化，灵活可靠，更少出错`
- 有时候系统开发者的角度与用户的角度会有矛盾。`用户的需求往往需要更复杂的构筑，这对程序员不友好。`

设备的模块化、层次化过多会使软件开销急剧增加。现在也有直接打通的从高层次直达低层次的设计。操作系统的实现并非一成不变，工程问题最终应该落于实际，根据当时情况做权衡。

微内核架构：
>![[lec2-OS.pdf#page=46]]

微内核的概念是相对于宏内核的。微内核是将宏内核的功能做压缩，将很多宏内核的功能分离出内核而变成用户态，仅保留一些基本的功能。这样能够增加安全性、更可扩展、更少代码、更可靠。
缺点在于内核越小，功能越不完善，内核调用功能减少会导致调用次数大幅增加，导致内核态和用户态的切换次数更频繁，使系统性能、效率下降。

现在的这种方案是部分分离出内核态，被称为混合内核模式。（包括Linux、鸿蒙、Android等）
