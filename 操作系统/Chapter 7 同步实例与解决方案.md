### 生产者、消费者问题(Bounder-Buffer Problem)
- 具有一个有限缓冲区(buffer)，生产者向其中放入数据、消费者从其中取出数据
- 分析问题：有多少限制条件？
	- ==操作系统一般解决方法：信号量(semaphore)==，一般有几个限制条件就几个信号量。
	- 限制条件1：互斥，每次有且仅有一个进程能进入该区域操作
	- 限制条件2：有限空间，当buffer full情况下不能再写入（针对producer）
	- 限制条件3：buffer empty情况下不能再读出销毁数据（针对consumer）

- 大小为n的buffer：
	- Semphore：mutex lock（互斥信号量）=1
	- Counting Semphore（计数信号量）：full=0
		- 表示缓存区数据数量
	- Counting Semphore（计数信号量）：empty=n
		- 表示空闲数据数量

```
# producer process 生产者进程
while (true) {
	...
	/* produce an item in next_produced 生产数据，非临界区 */
	...
	wait(empty); // empty--, 但是由于这个值很敏感, 一定要在锁函数里面执行
	wait(mutex);
	...
	/* add next produced to the buffer 加入数据，临界区 */
	...
	signal(mutex); // 互斥锁
	signal(full); // full++, 通知消费者可以消费的量增加
	
	// ==这个signal的位置可不可以在临界区前面？==
	// 不能。前面empty修改在临界区前是保证临界区内一定不溢出，这里full在临界区后同样也是在保证临界区一定不空。
	
}
```

```
# consumer process 消费者进程
while (true) {
	wait(full); // full--
	wait(mutex);
	...
	/* 临界区 */
	...
	signal(mutex);
	signal(empty); // empty++, 通知生产者可以生产的空位增加
	
	...
}
```

### Readers-Writers Problem
- 不对等问题
	- 读者只会读数据，不会改变数据
	- 写者既会读数据又会该数据
- 矛盾核心：
	- 写进程写入过程中排斥所有其他进程
	- 读进程可以一起读，不会影响
		- 保证读的时候一起读，提高效率
- 进一步分析限制条件：
	- 条件一：写入互斥锁（写进程会和其他读、写进程互相冲突）
		- `rw_mutex = 1`
	- 条件二：计数器，没人的时候第一个读者往里进，最后一个读者出门的时候释放锁
	- 条件三：count的值非常敏感，要保证read_count同步，保护这个变量。
		- `mutex = 1`

# \*Data Consistency 数据一致性问题
	在更大范围的数据一致性问题。
	相同的共享数据在不同的地方不匹配的问题。

- 根源：所有的系统都是共享数据的方式使用。

##### 基本性质
- 1.一致性
	- 所有数据都相同
- 2.可用性
	- 请求可以在短时间内被快速响应
- 3.分区容忍度
##### 帽子理论 CAP Theorem: 上述三个基本性质只能具备两个

##### 一致性边界
- OS/Networking 更偏向于可用性
- Database/Storage 更偏向于一致性

##### 核心问题：latency tradeoff 延迟容忍
- cap问题的核心在于延迟容忍。只要延迟够低，所有问题都可以容忍；只要容忍度够高，所有方案都可以采用。

##### 2PC：two phase commit protocol
- 第一阶段：协调者向所有节点发起投票（包括自己），等待结果
- 第二阶段：所有节点赞同时则通过，有节点冲突时就不通过

- 问题：如果协调者离线或者参与者离线怎么办？
	- 没有人收票，离线者的赞成与否情况不清楚。
	- blocking protocol：只要有人没投票（掉线、故障、投不了票），此时无法通过任何提案

- 2PC很安全，但活跃度（更新频率）很差。

##### 3PC：Three Phase Commit Protocol
- 改进：
	- 提出预先通知(precommit, do commit两个阶段)
	- 设置每个阶段的截止时间。超出截止时间认为票无效

- 第一阶段：协调者发起提案申请，节点回答。
- 第二阶段：协调者发起提案，节点承认。
- 第三阶段：协调者最终确认结果，节点回复。

- 问题：
	- 如果参与者的延时高于截止时间怎么办？
	- 在投票流程中，超时者上线并投出反对怎么办？

### Paxos
- 相似于2PC，但也有不同
	- 不特定的组织者
	- 让节点决定在几个并发的提案中接受哪一个，拒绝哪一个
	- 半数加一票通过制
		- 演化到区块链中
	- 多轮次投票

- 轮次内阶段
	- 阶段一：选举：是否认可leader
	- 阶段二：投票，是否认可value
	- 阶段三：发布结果，公布value

- 问题：
	- leader发生故障：启用新轮次
	- 前一轮投票未参与：无视前版本，投最新轮次的票，不投老票。
	- 永远保证在线的半数加一的节点赞成提案

- 后续发展中难以处理的问题：
	- 单人投票：不能接受。
	- 不满足半数加一：有两个节点票数相同且都不满足半数加一。
	- 由于网络延迟，不同提案在不同分区被通过了
	- ...

- Multi-Paxos
	- 尽可能减少投票次数
		- 同一张选票上容纳更多的选项。（减少节点间通讯）

- Raft
	- 选择版本数量最全的节点作为leader
